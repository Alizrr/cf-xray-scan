# cf-xray-scan

پیدا کردن آی‌پیِ کلودفلری که برای کانفیگِ *خودت* بیشترین سرعت آپلود را می‌دهد.

اگر یک کانفیگ VLESS / VMess / Trojan پشت کلودفلر داری، دامنه‌ی داخل کانفیگت از طریق
هر آی‌پی کلودفلر قابل دسترسی است — ولی از جایی که تو نشسته‌ای، بعضی از این آی‌پی‌ها
سریع‌اند و بیشترشان کند یا مرده. اسکنرهای معمولِ «آی‌پی تمیز» فقط لبه‌ی کلودفلر را
پینگ می‌کنند یا یک اندپوینت عمومی را تست می‌گیرند. این یکی فرق دارد: برای هر آی‌پی،
واقعاً تونلِ **کانفیگ خودت** را از روی همان آی‌پی بالا می‌آورد و سرعت آپلودِ واقعی را
از داخلش اندازه می‌گیرد. پس عددی که می‌بینی، همان عددی است که موقع وصل‌شدن خواهی داشت.

> نسخه‌ی انگلیسی: [README.md](README.md)

## چرا ساختمش

بیشتر اسکنرها به این جواب می‌دهند که «کدام آی‌پی کلودفلر به‌طور کلی سریع است؟». ولی
سؤالی که واقعاً برایم مهم بود این است: «کدام آی‌پی *برای کانفیگ من، از روی خط من،
همین الان* سریع است؟» — و این دوتا یکی نیستند. مخصوصاً آپلود: خیلی از آی‌پی‌هایی که
روی دانلود خوب به‌نظر می‌رسند، روی آپلود می‌مانند. برای همین ابزار هر آی‌پی را با
SNI/host/path واقعی‌ات از طریق xray می‌زند و چند مگابایت از داخلش آپلود می‌کند. بدون حدس.

## پیش‌نیازها

- پایتون ۳.۸ به بالا (فقط کتابخانه‌ی استاندارد — نیازی به `pip install` نیست)
- فایل اجرایی `xray`. اگر v2rayN داری، همین الان داخلش هست، مثلاً
  `...\v2rayN-windows-64\bin\xray\xray.exe`. وگرنه از
  [ریلیزهای Xray-core](https://github.com/XTLS/Xray-core/releases) بگیر.
- فایل `xray_test.py` باید کنار `cf_xray_scan.py` باشد (اسکریپت آن را import می‌کند).

## استفاده

اجرای ساده (رنج‌های پیش‌فرض کلودفلر، ۶۰ آی‌پی نمونه):

```powershell
python cf_xray_scan.py "<لینک کانفیگ>" --xray "C:\path\to\xray.exe"
```

اسکن دقیق‌تر روی رنج‌هایی که معمولاً بهترند:

```powershell
python cf_xray_scan.py "<لینک کانفیگ>" `
  --xray "C:\path\to\xray.exe" `
  --cidr 104.16.0.0/13 --cidr 172.64.0.0/13 `
  --sample 100 --concurrency 6 --ready-timeout 12
```

از روی لیست آی‌پی خودت (هر خط یک آی‌پی یا CIDR):

```powershell
python cf_xray_scan.py "<لینک کانفیگ>" --xray "C:\path\to\xray.exe" --ip-file ips.txt
```

لینک کانفیگ را حتماً داخل گیومه بگذار — چون `&` دارد و شل وگرنه آن را می‌خورد.

## خروجی چیست

- یک جدول مرتب‌شده از بهترین آی‌پی‌ها (آپلود + پینگ) در ترمینال
- فایل `cf_xray_results.csv` — نتیجه‌ی کامل
- فایل `best_configs.txt` — لینک‌های آماده که آی‌پی برنده توشان جایگزین شده. کل فایل را
  کپی کن و در v2rayN از مسیر *Add configs from clipboard* وارد کن.

بعد کافی است کلاینت را روی بهترین آی‌پی بگذاری (یا `best_configs.txt` را ایمپورت کنی)؛
SNI و host و path دقیقاً همان قبلی می‌مانند.

## گزینه‌ها

| فلگ | کارش | پیش‌فرض |
|------|------|---------|
| `--cidr` | یک رنج CIDR، قابل تکرار | رنج‌های رسمی کلودفلر |
| `--ip-file` | فایل آی‌پی‌ها / CIDRها | — |
| `--xray` | مسیر فایل xray | کنار اسکریپت / روی PATH |
| `--sample` | چند آی‌پی تست شود | ۶۰ |
| `--concurrency` | چند تونل هم‌زمان | ۸ |
| `--upload-mb` | حجم آپلود هر آی‌پی، مگابایت | ۴ |
| `--ready-timeout` | مهلت بالا آمدن تونل | ۶ ثانیه |
| `--make-links` | برای چند آی‌پی برتر لینک بسازد | ۵ |
| `--top` | چند ردیف چاپ شود | ۱۵ |

یک اسکریپت کوچک‌تر هم هست — `xray_test.py "<لینک>" --xray ...` — که فقط یک کانفیگ را
وصل می‌کند و پینگ / دانلود / آپلود می‌دهد. برای یک چک سریعِ «اصلاً این کانفیگ زنده هست؟»
به‌دردبخور است.

## چند نکته‌ی صادقانه

- پینگ بالا (۹۰۰ms+) و سقفِ آپلودِ پایین معمولاً به‌خاطر مسیر یا سرورِ origin است، نه
  اسکنر. این ابزار واقعیت را اندازه می‌گیرد؛ نمی‌تواند از آن جلو بزند.
- اگر تعداد زیادی `tunnel did not come up` دیدی، `--ready-timeout` را ببر روی ۱۰ تا ۱۵
  و `--concurrency` را کم کن — وقتی تأخیر بالاست، تعداد زیاد تونلِ هم‌زمان فقط روی هم
  می‌افتند.
- این فقط وقتی معنا دارد که کانفیگ واقعاً پشت کلودفلر باشد. اگر دامنه مستقیم به origin
  ریزالو شود، هیچ آی‌پی کلودفلری به آن نمی‌رسد و صفر آی‌پیِ کارا می‌گیری (که خودش یک
  اطلاعات مفید است).
